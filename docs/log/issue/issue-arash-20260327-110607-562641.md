## Unified PDF ingestion: common pool + browser action + OA fetch + needs-PDF queue

### Problem

Current multi-project PDF ingest workflow is fragmented and slow:
1. Find paper online → download PDF manually (often behind OAuth/SSO)
2. Drop into Zotero → normalize filename → export .bib → run `biblio bibtex fetch-pdfs` + `biblio bibtex merge` → datalad save + push
3. On HPC: `datalad update --merge` (often 2-pass due to .bib merge conflicts) → verify

Each project maintains its own PDF copies. Same paper in 3 projects = stored 3 times. No way to leverage lab-wide PDF acquisition.

### Proposed design: two-tier model

**Tier 1 — Common library** (e.g. `/storage/share/sirocampus/bib/` or `~/bib/`)
- Full biblio workspace: owns PDFs, canonical BibTeX, docling/grobid output, RAG
- Single source of truth; deduplication across all projects and lab members
- Drop folder watched by daemon → auto-ingests new PDFs

**Tier 2 — Project workspaces** (e.g. `pixecog/bib/`)
- Owns citekey selection (which papers this project cares about)
- PDFs are symlinked from Tier 1, never copied
- PDFs stop travelling through datalad; only `.bib` text files sync

### Two ingestion paths

**Pull path (biblio-initiated)** — triggered when a citekey is added to a project srcbib:
1. Check common pool → symlink, done
2. Check OpenAlex `open_access.oa_url` (Unpaywall) → auto-fetch, done
3. Optionally check forward/back citation graph (already in `biblio graph expand`) for pre-emptive pool population
4. If blocked → add to "needs PDF" queue

**Push path (user-initiated)** — browser action:
- **Bookmarklet** (zero install): JS snippet, one click on any publisher page → downloads PDF using existing browser SSO session → POSTs to local `biblio` daemon → lands in common pool
- **Browser extension** (enhanced): popup shows citekey match, which projects reference this paper, queue status; injects "Add to biblio" button on publisher pages
- Both fall back to saving to `~/bib-inbox/` (watched drop folder) if daemon is not running
- Auth is handled by the browser's existing institutional SSO — no credential storage in biblio

### Needs-PDF queue (first-class concept)

```
biblio queue list          # papers needing PDFs, with OA status
biblio queue open <key>    # open paper URL in browser (extension picks it up)
biblio queue drain         # re-run OA fetch attempts for all queued items
```

Queue lives in `bib/config/fetch_queue.yml` — text only, datalad-syncable, visible on dashboard. The queue is the bridge between pull and push: biblio surfaces what's missing, the browser action is how the user fulfills it.

### Key benefits
- Eliminates Zotero from the workflow entirely
- PDFs acquired once for the whole lab, never duplicated across projects
- Auth-aware PDF acquisition without storing credentials
- OA papers fetch automatically; paywalled papers route to user via queue + browser action
- datalad sync scope reduced to `.bib` text files only — merge conflicts become trivial

### Related
- `biblio/src/biblio/pdf_fetch.py` — extend to check common pool before network
- `biblio/src/biblio/library.py` — currently per-project; needs global/common library concept
- `biblio/src/biblio/graph.py` — `expand_openalex_reference_graph` could pre-populate queue
- OpenAlex `open_access` field already resolved in `openalex.py`
