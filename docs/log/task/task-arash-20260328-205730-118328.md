# GUI: Backend API endpoints for new features

## Goal

Add FastAPI endpoints in `src/biblio/ui.py` for all new backend features so the React frontend can access them. This is the prerequisite for all frontend GUI tasks.

## Context

- `src/biblio/ui.py` is a large FastAPI app with inline component serving
- Existing pattern: `@app.get`/`@app.post` endpoints at `/api/...`
- Background actions use a thread + status polling pattern (see docling-run, grobid-run, openalex-resolve as examples)
- New backend modules to expose: `tag_vocab`, `autotag`, `summarize`, `concepts`, `compare`, `reading_list`, `cite_draft`, `lit_review`, `present`, `query`, updated `collections`
- `/api/model` returns the full data model ‚Äî needs derivative flag extensions
- PaperTab subtabs are `["PDF", "Markdown", "Figures", "Refs"]` ‚Äî new subtabs will need content endpoints
- Collections API at `/api/collections` needs smart collection support

## Prompt

> Call `project_context()` and `runtime_conventions()` to understand the workspace.
>
> Read `src/biblio/ui.py` thoroughly to understand:
> - How `/api/model` assembles the data payload (what fields each paper dict has)
> - How background actions work ‚Äî study the `docling-run` and `grobid-run` patterns: they use a thread, a status dict, and a `/status` polling endpoint
> - How `/api/collections` CRUD works
> - How `/api/library/{citekey}` updates work
> - How file-serving endpoints work (`/api/files/pdf/`, `/api/files/docling/`)
>
> Then read the public APIs of: `tag_vocab.py`, `autotag.py`, `summarize.py`, `concepts.py`, `compare.py`, `reading_list.py`, `cite_draft.py`, `lit_review.py`, `present.py`, `query.py`, `collections.py`.
>
> Add these API endpoints to `ui.py`, following existing code style exactly:
>
> ### Tag vocabulary
> - `GET /api/tag-vocab` ‚Äî returns vocabulary from `tag_vocab.load_tag_vocab()`
> - `POST /api/library/lint` ‚Äî runs `tag_vocab.lint_library_tags()`, returns issues as JSON
>
> ### Auto-tagging (background action pattern)
> - `POST /api/actions/autotag` ‚Äî body: `{citekey?, tiers?, model?}`. Runs autotag in background thread.
> - `GET /api/actions/autotag/status`
> - `POST /api/actions/autotag/cancel`
>
> ### Summaries (background action pattern)
> - `GET /api/papers/{citekey}/summary` ‚Äî returns summary markdown content if exists, 404 if not
> - `POST /api/actions/summarize` ‚Äî body: `{citekey, force?, model?}`. Background thread.
> - `GET /api/actions/summarize/status`
>
> ### Concepts
> - `GET /api/papers/{citekey}/concepts` ‚Äî returns concepts YAML content if exists
> - `GET /api/concepts/index` ‚Äî returns cross-paper concept index
> - `POST /api/concepts/search` ‚Äî body: `{query}`. Returns matching concepts and citekeys.
> - `POST /api/actions/concepts-extract` ‚Äî body: `{citekey?, all?}`. Background thread.
> - `GET /api/actions/concepts-extract/status`
>
> ### Comparison (background action)
> - `POST /api/actions/compare` ‚Äî body: `{citekeys, dimensions?, model?}`. Background thread.
> - `GET /api/actions/compare/status`
> - `GET /api/comparisons` ‚Äî list existing comparison files
>
> ### Reading list
> - `POST /api/reading-list` ‚Äî body: `{question, count?, model?}`. Can use background action pattern.
> - `GET /api/actions/reading-list/status`
>
> ### Citation drafting
> - `POST /api/cite-draft` ‚Äî body: `{text, style?, max_refs?, model?}`. Background action.
> - `GET /api/actions/cite-draft/status`
>
> ### Literature review
> - `POST /api/actions/review-query` ‚Äî body: `{question, model?}`. Background action.
> - `POST /api/actions/review-plan` ‚Äî body: `{seed_citekeys, question?, model?}`. Background action.
> - `GET /api/actions/review/status`
>
> ### Presentations (background action)
> - `GET /api/papers/{citekey}/slides` ‚Äî returns slides markdown if exists
> - `POST /api/actions/present` ‚Äî body: `{citekey, template?, model?}`. Background thread.
> - `GET /api/actions/present/status`
>
> ### Smart collections
> - Update `POST /api/collections` to accept optional `query` field
> - Add `PATCH /api/collections/{col_id}/query` ‚Äî update smart collection query
> - Update `GET /api/collections` to include resolved smart collection membership counts
>
> ### Model payload extension
> - Extend `/api/model` per-paper data to include boolean flags: `has_summary`, `has_concepts`, `has_slides`, `has_autotag` (check for file existence in derivatives dirs)
>
> After all changes, run `make test`. Then use `datalad_save` MCP tool to commit with message: "Add backend API endpoints for GUI upgrade features".
